# Zekelman Standard 3 Teacher-Reviewed Diagnostic Planning Grid

Current branch: `feature/standard-3-source-verification-final-decisions`
Status: documentation and planning support only
Runtime status: `not_runtime_ready`
Question-generation status: `not_question_ready`

## 1. Purpose

This grid summarizes the recorded teacher-reviewed decisions for Zekelman Standard 3 so they can guide future diagnostic planning. It is meant to help teachers, school leaders, and developers see which strands may be considered for a later diagnostic blueprint and which boundaries must remain protected.

This grid does not create new reviewer decisions. It only restates decisions and notes already recorded in the Standard 3 review materials.

## 2. Hard Boundaries

- This is not runtime-ready.
- This is not question-ready.
- This does not authorize student-facing questions.
- This does not modify reviewed-bank content.
- This does not activate any skill in the app.
- This does not create answer keys or active question templates.
- This does not override Zekelman 2025 source wording or teacher source-verification requirements.

## 4. MVP Diagnostic Planning Recommendation

The first diagnostic blueprint should focus only on these planning lanes:

- `3.01`: taught noun recognition, translation, and controlled multiple-meaning work.
- `3.02`: simple shoresh identification only.
- `3.05`: split pronoun referent tracking and pronominal suffix decoding.
- `3.06`: visible prefixes, articles, and basic prepositions only.
- `3.07`: foundational tense, person, and form clues only.

These strands should remain cautious or later:

- `3.04`: use only with wording revision, separate noun/adjective and סמיכות lanes, and source verification.
- `3.08`: use only with wording revision, context protection, and source/PDF verification.
- `3.10`: defer to a later phase and keep out of the MVP diagnostic blueprint except as a caution note.

## 5. Remaining Blockers Before Question Generation

Before any question generation, the project still needs:

- Source/PDF verification checks completed.
- Teacher-approved wording boundaries.
- An explicit diagnostic blueprint file.
- Explicit skill-to-question-type mapping.
- A separate promotion gate allowing question generation.

Until those steps happen, these decisions remain planning guidance only.

## 6. Next Recommended Development Step

After this grid is reviewed, the next recommended development step is to create the Standard 3 MVP diagnostic blueprint as a non-runtime, review-only planning artifact. That blueprint should preserve the boundaries above and should not create student-facing questions or activate any skill in the app.
